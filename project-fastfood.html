Designing a Fast-Food System in Python

Python OOP pandas CLI
💻 View the code on GitHub
Problem

Turning spreadsheet rows into objects that know what they are

Most real systems start out looking like a pile of spreadsheets: a list of cashiers, a list of customers, a handful of product catalogs. The design problem is what happens next — how do you turn each row into something that carries its own behavior, so the rest of the program never has to ask "wait, what kind of row is this?" This project is a small fast-food ordering system built to practice exactly that: reading six CSV files that stand in for a database, and modeling everything they contain — cashiers, customers, hamburgers, sodas, happy meals — as proper objects, built with abstract base classes, inheritance, and composition rather than a wall of if/elif statements.

Method

Abstract base classes, not conditionals

Every core concept — Product, FoodPackage, User — is defined as an abstract base class with the methods every subclass is required to implement. Concrete types then supply their own behavior: a Hamburger knows its own type and packages itself in a paper wrap, a Soda comes in a bottle, a Cashier and a Customer each describe themselves differently despite sharing a common User parent. Every object in the system can describe itself — no external code needs to know its internals to print it.

Product is an abstract base class; each concrete type pairs itself with a FoodPackage
class Product(ABC): def __init__(self, id: str, name: str, price: float): self.id = id self.name = name self.price = price def describe(self): return f"Product - Type: {self.type()}, Name: {self.name}, ...{self.foodPackage().describe()}." # every subclass must implement these two @abstractmethod def type(self) -> str: pass @abstractmethod def foodPackage(self) -> FoodPackage: pass class Hamburger(Product): def type(self) -> str: return "Hamburger" def foodPackage(self) -> FoodPackage: return Wrapping()

From CSV rows to typed objects

Six CSV files stand in for a database. A CSVFileManager loads each one into a pandas DataFrame, and a small hierarchy of Converter classes turns every row into the right kind of object — detecting a product's type straight from its id prefix. From that point on, nothing downstream ever touches a raw row again; the rest of the program only deals in real Cashier, Customer, and Product instances.

Why this matters beyond the practice

Turning raw rows — from a CSV, a database query, or an API request — into well-defined objects with clear responsibilities is one of the most common patterns in production backend code. The same shape shows up everywhere: an ORM converting database rows into model instances, a serializer converting a JSON payload into a typed object, a message consumer converting a queue payload into a domain event. Getting comfortable with abstraction, inheritance, and composition at this small scale is what makes that code readable at a much larger one.
